SET09102 Coursework: API Reference
API Overview

Base URL: https://set09102-api.b-davison.workers.dev

Authentication: JWT Bearer token (required for most endpoints)

Content Type: application/json

Response Format: JSON
💡 Interactive Docs: A Swagger UI with live try-it-out is available at https://set09102-api.b-davison.workers.dev/ (the root path). No SSH tunnel or special setup required — simply point your HttpClient at the base URL above.
1. Authentication Endpoints
POST /auth/register

Create a new user account.

Authentication Required: No

Request Body:

{
  "firstName": "John",
  "lastName": "Doe",
  "email": "john@example.com",
  "password": "SecurePassword123!"
}

Validation Rules:

    firstName: Required, 1-50 characters
    lastName: Required, 1-50 characters
    email: Required, valid email format, unique
    password: Required, minimum 8 characters, must contain uppercase, lowercase, number, and special character

Success Response (201 Created):

{
  "id": 123,
  "email": "john@example.com",
  "firstName": "John",
  "lastName": "Doe",
  "createdAt": "2026-02-03T10:00:00Z"
}

Error Response (400 Bad Request):

{
  "error": "Validation failed",
  "message": "Email already exists"
}

2. Item Discovery Endpoints
GET /items

List all available items (with optional filters).

Authentication Required: No

Query Parameters:

    category (string, optional): Filter by category slug (e.g., "tools", "camping")
    search (string, optional): Search in title and description
    page (integer, optional): Page number (default: 1)
    pageSize (integer, optional): Items per page (default: 20, max: 100)

Example: GET /items?category=tools&page=1&pageSize=20

GET /items/nearby

Find items near a location using PostGIS spatial queries.

Authentication Required: No

Query Parameters:

    lat (float, required): Latitude (-90 to 90)
    lon (float, required): Longitude (-180 to 180)
    radius (float, optional): Search radius in kilometers (default: 5, max: 50)
    category (string, optional): Filter by category slug

Example: GET /items/nearby?lat=55.9533&lon=-3.1883&radius=10&category=tools

Success Response (200 OK):

{
  "items": [
    {
      "id": 789,
      "title": "Electric Drill",
      "description": "Cordless drill with battery",
      "dailyRate": 5.00,
      "categoryId": 1,
      "category": "Tools",
      "ownerId": 456,
      "ownerName": "Sarah Smith",
      "latitude": 55.9600,
      "longitude": -3.1900,
      "distance": 2.3,
      "isAvailable": true,
      "averageRating": 4.5,
      "imageUrl": "https://example.com/images/drill.jpg"
    }
  ],
  "searchLocation": {
    "latitude": 55.9533,
    "longitude": -3.1883
  },
  "radius": 10.0,
  "totalResults": 1
}

💡 PostGIS Implementation: This endpoint uses PostGIS ST_DWithin function for efficient spatial queries. The distance field is calculated in kilometers using ST_Distance on geography types.

Error Response (400 Bad Request):

{
  "error": "Validation failed",
  "message": "Invalid coordinates: latitude must be between -90 and 90"
}

3. Rental Workflow Endpoints
POST /rentals

Initiate a rental request (Borrower side).

Authentication Required: Yes

Headers:

Authorization: Bearer {token}
Content-Type: application/json

Request Body:

{
  "itemId": 789,
  "startDate": "2026-02-10",
  "endDate": "2026-02-12"
}

Validation Rules:

    itemId: Required, must exist and be available
    startDate: Required, must be today or future date
    endDate: Required, must be after startDate
    Cannot rent your own item
    Item must not have conflicting approved rentals for these dates

Success Response (201 Created):

{
  "id": 555,
  "itemId": 789,
  "itemTitle": "Electric Drill",
  "borrowerId": 123,
  "borrowerName": "John Doe",

}

{

PATCH /rentals/{id}/status

Update rental status (Approve, Reject, Return, Complete).

Authentication Required: Yes (permissions depend on status transition)

Headers:

Authorization: Bearer {token}
Content-Type: application/json

URL Parameters:

    id (integer): Rental ID

Request Body:

{
  "status": "Approved"
}

Valid Status Transitions:
From Status 	To Status 	Who Can Perform 	Notes
Requested 	Approved 	Owner only 	Owner accepts the rental request
Requested 	Rejected 	Owner only 	Owner declines the rental request
Approved 	Out for Rent 	System or Owner 	Automatically on start date, or manually by owner
Out for Rent 	Overdue 	System 	Automatically if not returned by end date
Out for Rent 	Returned 	Borrower 	Borrower marks item as returned
Overdue 	Returned 	Borrower 	Late return
Returned 	Completed 	Owner 	Owner confirms item is in good condition

Success Response (200 OK):

{
  "id": 555,
  "status": "Approved",
  "updatedAt": "2026-02-04T09:00:00Z"
}

Error Response (400 Bad Request):

{
  "error": "Invalid state transition",
  "message": "Cannot transition from Approved to Rejected"
}

Error Response (403 Forbidden):

{
  "error": "Forbidden",
  "message": "Only the owner can approve rental requests"
}

💡 State Machine Implementation: Consider implementing the State Pattern to handle these transitions cleanly. See the coursework specification for examples.

5. Error Handling

All API errors follow a consistent format:
Standard Error Response

{
  "error": "Error type",
  "message": "Human-readable error message",
  "details": {
    "field": "Additional error details (optional)"
  }
}

HTTP Status Codes
Status Code 	Meaning 	When Used
200 OK 	Success 	Successful GET, PUT, PATCH requests
201 Created 	Resource created 	Successful POST requests
400 Bad Request 	Validation error 	Invalid input, validation failures
401 Unauthorized 	Authentication required 	Missing or invalid JWT token
403 Forbidden 	Permission denied 	User lacks permission for action
404 Not Found 	Resource not found 	Item, rental, or user doesn't exist
409 Conflict 	Resource conflict 	Duplicate email, overlapping rentals
500 Internal Server Error 	Server error 	Unexpected server-side issues

8. Rate Limiting & Best Practices
⚠️ Rate Limiting:

    Maximum 100 requests per minute per user
    Maximum 1000 requests per hour per user
    Exceeded rate limits return 429 Too Many Requests

Best Practices

    Cache Responses: Cache category lists and user profiles locally
    Pagination: Always use pagination for list endpoints
    Error Handling: Always handle network errors and API errors gracefully
    Token Management: Store JWT tokens securely using SecureStorage
    Location Precision: Use appropriate precision for coordinates (6 decimal places)
    Date Formats: Always use ISO 8601 format (yyyy-MM-dd) for dates
    Base URL: Use https://set09102-api.b-davison.workers.dev as the base URL in your HttpClient

💡 Performance Tip: Use the nearby endpoint instead of fetching all items and filtering client-side. The PostGIS spatial index makes this much faster.
9. Common Issues & Troubleshooting
Connection Refused / Timeout

Problem: HttpRequestException: Connection refused or timeout

Solution:

    Confirm you are using https://set09102-api.b-davison.workers.dev as the base URL (no SSH tunnel needed)
    Check your internet connection
    The API is globally available via Cloudflare Workers — no university VPN or tunnel required

401 Unauthorized

Problem: 401 Unauthorized on authenticated endpoints

Solution:

    Check that JWT token is being sent in Authorization header
    Verify token hasn't expired (tokens expire after 7 days)
    Re-login to get a new token

400 Bad Request with Validation Errors

Problem: 400 Bad Request with validation message

Solution:

    Check request body matches API specification exactly
    Verify data types (e.g., dailyRate should be decimal, not string)
    Check date formats (must be yyyy-MM-dd)
    Ensure required fields are present

PostGIS Query Returns No Results

Problem: GET /items/nearby returns empty list

Solution:

    Verify latitude/longitude are correct (lat: -90 to 90, lon: -180 to 180)
    Increase search radius (try 50km)
    Check that items exist in the database with valid locations
